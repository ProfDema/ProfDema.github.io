Assignment 1: Standard Input/Output and Memory Allocation

Introduction

For this assignment, you'll write a C program that manipulates sounds by adding sound effects to them. Namely, the effects you will create are listed here:

  • fade_in simulates the effect of a sound approaching from a distance to the observer.
  • fade_out simulates the effect of a sound moving away from the observer
  • pan simulates the effect of a sound panning from the left to the right of the observer

Completing this assignment will require you to use command-line options, standard I/O, string functions, and memory allocation in C. You will also use piping and output redirection in the Bash shell.

Sounds are waves of air pressure. When a sound is generated, a sound wave consisting of compressions (increases in pressure) and rarefactions (decreases in pressure) moves through the air. This is similar to what happens if you throw a stone into a pond: the water rises and falls in a repeating wave.

When a microphone records sound, it takes a measure of the air pressure and returns it as a value. These values are called samples and can be positive or negative corresponding to increases or decreases in air pressure. Each time the air pressure is recorded, we are sampling the sound. Each sample records the sound at an instant in time; the faster we sample, the more accurate is our representation of the sound. The sampling rate refers to how many times per second we sample the sound. For example, CD-quality sound uses a sampling rate of 44100 samples per second; sampling someone's voice for use in a VOIP conversation uses far less than this. Sampling rates of 11025 (voice quality), 22050, and 44100 (CD quality) are common.

For mono sounds (those with one sound channel), a sample is simply a positive or negative integer that represents the amount of compression in the air at the point the sample was taken. For stereo sounds, a sample is actually made up of two integer values: one for the left speaker and one for the right.

A WAV file contains a header of 44 bytes that contains data about the sampling rate, total number of sound samples, and a few other pieces of information: 

// Define WAV file format header

    char riff[4];     // "RIFF", 4 bytes
    int total_size;   // Total Size, 4 bytes
    char fmt[8];      // "WAVEfmt", 8 bytes
    int fmt_size;     // fmt chunk size, 4 bytes
    short format;     // type of format, 2 bytes
    short channel;    // channel, 2 bytes
    int rate;         // sampling rate, 4 bytes
    int avgbyte;      // rate * block, 4 bytes
    short block;      // channels * bit / 8, 2 bytes
    short bit;        // bits, 2  bytes
    char data[ 4 ];   // = "data", 4 bytes
    int data_size;    // data size, 4 bytes

See this page for a more detailed explanation of the values contained in the WAV header.

The remainder of the WAV file contains a contiguous sequence of int values, each representing a sound sample.

For example, the provided WAV file crow.wav has a data_size of 60416 bytes (you can tell also by looking at the file size and subtracting 44 bytes of the header) and a rate of 22050.

We see, for example, the number at offset 0x28 (i.e., at the 40th byte in the file) is 0x00ec0000. According the data description given above, at this location one finds data_size. Data size is an integer, which occupies 4 bytes, and it is written in little-endian which means that the real data size is 0x0000ec00 or, converted to a decimal number, 64016, as expected.

Note: Briefly, endianness dictates the order in which the bytes of a multi-byte number are stored in memory. For example, the number 168496141 would be printed as 0x0a0b0c0d in big-endian (i.e., most significant byte first), or 0x0d0c0b0a in little-endian (i.e., least significant byte first).

The first measured sound sample is at offset 0x2c, and is also in little-endian, so the actual sample value is 0xffb9, converted to decimal, that would be -71. A basic manipulation of sound samples, would consist of increasing or decreasing (scaling) the volume.

Here is how volume scaling works: multiplying each sample value by a number greater than 1 increases the volume; multiplying each sample by a number between 0 and 1 (exclusive) decreases the volume. For example, multiplying everything by 2 doubles the volume; multiplying everything by 0.5 halves the volume.

Reading the data in your program.

We will first pipe a hex dump of the WAV file (generated with the xxd program) to your program, which will process the data and output it to stdout, which will once more be piped to the xxd program to revert the hex dump back into a binary file and output the result into a WAV file as follows: 

xxd -g1 file1.wav | ./effects -param value | xxd -r >file2.wav

You may look up the usage instructions for xxd by checking man xxd or reading the online instructions here.

Your code must read the input in hex format from stdin, using scanf, and it must write the processed output in hex format imitating precisely the output of xxd. For example, consider the first recorded sample in the data above. It reads b9 ff. Its decimal value (as a short value is -71 (hint: keep in mind endianness!). Increasing the volume twice, means simply doubling this number, namely -71*2=-142. Next, we need to convert it to hexadecimal, and reverse the order of bytes due to little-endianness. The value your program should output for this sample is 72 ff.

Fade-in and Fade-out

Fade-in

Before we discuss how to accomplish fade-in, let's get acquainted with some fading-in. Listen to this monotonous sound of water bubbling. The volume is stable throughout. Now, running the following command: 

xxd -g1 water.wav | ./effects -fin 2000 | xxd -r > water_shortfade.wav

should produce the WAV file water with a short fade-in. Notice how the water linearly fades in over the first two seconds, then remains at maximum volume throughout. (2000 milliseconds corresponds to two seconds, or 88200 samples, because we're using sounds recorded at 44100 samples per second.)

How many milliseconds should we call fade_in with, in order to get water with a long fade-in, basically for the full length of the recording? The fade-in is slowly and linearly applied over the entire duration of the sound, so that the maximum volume is reached only at the very last sample.

To apply a fade-in to a sound, we multiply successive samples by larger and larger fractional numbers between 0 and 1. Multiplying samples by 0 silences them, and multiplying by 1 (obviously) keeps them the same. Importantly, multiplying by a factor between 0 and 1 scales their volume by that factor.

Here's an example. Assume we would like to apply fade-in over the first four milliseconds. Assuming the sampling rate is 1000 samples/sec. We then need to look at the first 4 samples (0.004 seconds * 1000 samples/second = 4 samples), numbered 0 to 3 in the table below for illustrative purposes. Both channels of these samples should be multiplied by the following factors to generate the fade-in:

Sample Number Multiply By...
0 0.0
1 0.25
2 0.5
3 0.75
>3 1 (unchanged)

Note that this example uses small numbers for the purpose of easy illustration—it is unlikely that you could hear any difference between the original file and the edited version with a 4-millisecond fade-in.

Fade-out

xxd -g1 rain.wav | ./effects -fout 2000 | xxd -r > rain_shortfade.wav

This time, the fade is a fade-out (from loud to quiet), and the fade-out begins fade_length milliseconds from the end of the sound rather than from the beginning.

Let's use a raining sound to demonstrate. As with the water bubbling above, The volume is stable throughout. Now, running the code above, we get rain with a short fade-out. The first few seconds of the rain are as before. Then, two seconds before the end, the fade-out starts, with the sound progressing toward zero volume. The final sample of the sound has value 0.

The multiplicative factors for fade out are the same as for fade in, but are applied in the reverse order. For example, if fade_length were 4 milliseconds, we would need to transform both the left and right channels of the last 4 samples (0.004 seconds * 1000 samples/sec = 4 samples) by multiplying them by 0.75, 0.5, 0.25, and 0, respectively. (Again, it is unlikely that you would be able to hear the difference with a 4ms fade-out time.)

Panning from Left to Right

xxd -g1 small2chan.wav | ./effects -pan 4 | xxd -r > small_pan.wav

This function takes a sound object and an integer indicating the number of milliseconds to which the pan will be applied. The files used above are here and here.

Let's listen to what panning sounds like. Here's an airplane sound. The entire sound is centred, and does not move in the stereo field as you listen. Now, running the code above, we get this airplane panning from left to right sound. The sound starts completely at the left, then slowly moves to the right, reaching the extreme right by the final sample.

Getting a sound to move from left to right like this requires a fade-out on the left channel and a fade-in on the right channel.

Here's an example. Assume we want to achieve pan in 4 milliseconds. This would require us to transform the first four samples (0.004 seconds * 1000 samples/sec = 4 samplse) of both the left and right channels. The following table indicates the factors by which the channels of these samples should be multiplied:

Sample Number Multiply Left Channel By... Multiply Right Channel By...
0 0.75 0
1 0.5 0.25
2 0.25 0.5
3 0 0.75
>3 0 1 (unchanged)

To understand how this function works, it might help to think of changing the volume using two volume controls: one for the left channel and one for the right. To make the sound seem like it's moving from left to right, you slowly lower the volume in the left ear and raise the volume in the right ear. There is no copying going on between the two channels. And for the record, this technique only works when corresponding samples of both channels are the same: experiment with this dog and lake sound to see what happens when channels contain different sounds.

Comparing Binary Files

For full marks, the WAV files created by your programs must match our sample WAV files exactly. WAV files are binary files, and a useful Unix utility for comparing binary files is cmp. Read the man page for cmp, especially its -l option. This can tell you whether two files differ and, if they do, the bytes in the files that do not match. Note that cmp -l outputs byte offsets in decimal but byte values themselves in octal (find a converter online if you do not know how to convert from octal to decimal).
