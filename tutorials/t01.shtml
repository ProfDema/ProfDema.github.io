Tutorial 1: The Shell

Introduction

The purpose of this lab is to practice using the shell to navigate a linux file system, to complete a few simple tasks, and to submit your files using git.

You may complete this lab on your own machine, if you are running linux, macOS, or using the bash shell on Windows 10, or you may complete the lab on a linux machine, either by logging into the machine directly or by using ssh. You probably want to complete the lab on the machine you'll be using while writing code for this course. However, there's an advantage to doing so on the lab machines: your code will be tested on a lab machine, so you should always test your assignment code on a lab machine before submission. This doesn't mean you have to work on campus: the lab machines can be accessed remotely by sshing to dhXXXXpcYY.utm.utoronto.ca where XXXX is the lab room number (2020 or 2026, in Deerfield Hall) and YY is the PC number (01 to 30).

If you're using your own machine, you'll need to install a number of core development tools, like the C compiler. On many linux systems and on the Windows 10 bash shell, you can install them by typing sudo apt update (don't skip this) followed by sudo apt install build-essential on the command line. Note that this command uses sudo, so you need to have root (i.e., administrator) permissions.

If you are a macOS user, you should be aware that there will be a number of major differences between your environment and that in the lab. For example, recent versions of macOS no longer use the Bash shell. Moreover, the gcc command on macOS is actually aliased (explained later in this tutorial) to llvm rather than the gcc compiler. Therefore, you may observe differences in errors and warnings that are generated. While it may be useful to test your code on multiple compilers (e.g., llvm has developed a reputation for having more helpful error/warning messages), any tutorials or assignments that you submit throughout the course must compile and run correctly on the officially-supported lab environment. Moreover, please keep it under consideration that most of us will not be familiar with macOS, so can not guarantee any help/support, but you may have TAs and/or classmates familiar with macOS who are willing to offer help.

Creating Aliases and Persistent Shell Configuration

I find it fairly useful to have ls always show me all files, so I have it set as an alias. Since I prefer to see the file sizes in human-readable units (e.g., 1K, 234M, 2G, instead of displaying all file sizes in bytes), I also use the h flag. Type alias ls='ls -lah' in the shell. Then, try ls (without arguments) again. See how it acts? alias creates a replacement; when you type ls, the shell replaces it with ls -lah.

Now, exit (by typing exit) and then reopen a terminal. Try ls again. See how it's changed back so that it only provides basic information? We need a way to make this change permanent.

To do this, we'll need to make certain configuration files run when we log into the terminal. To create these files, you'll need to edit text files. Three of the most popular UNIX text editors are nano (very basic features), emacs, and vim (these latter two are much more powerful, but have a steeper learning curve). You may be amused to learn that the emacs vs. vim debate sparked many "flame wars" between users in the 1980s; check out this Wikipedia page on the "Editor war" for more historical background. There are many tutorials available online for both emacs and vim (for nano, you shouldn't really need a tutorial). Some more unique tutorials for vim are VIM Adventures and Interactive Vim tutorial. I highly recommend that you get used to using a UNIX text editor in this course, since it will help you stick to working in the terminal and master linux much more quickly.

Use your text editor to edit a file in your home directory. The file should be named .bash_profile. Put this text into that file: 

if [ -f ~/.bashrc ]; then
   source ~/.bashrc
fi

That says, "If there is a file named .bashrc in my home directory, then source it." Note that ~ (tilde) is used as shorthand for "my home directory" and that source is a command: it means "run this script and save any variables it creates in my environment."

Now, anything you put into your .bashrc should run when you log in, as if you had typed each command into the command line yourself. You might end up putting a lot into this file: you could update your path, add aliases, change your prompt, etc. For now, let's add an alias. Edit the .bashrc file in your home directory. Add this line to it: 

alias ls='ls -lah'

To test that it works, exit your terminal and then start it again. When you type ls, you should see the same detailed output that you saw when you typed ls -lah.

If that works, then let's do one more thing: let's change our prompt. In the command line, type PS1='Hi!'. What just happened? Try this: PS1='\w\$ '. To see why \w is your current working directory, check out this page in the Bash Reference Manual. Then, set your prompt to something you like permanently by assigning to PS1 in your .bashrc. Note: PS1 is an environment variable, just like the PATH environment variable discussed in the Week 1 slides.

man

We should pause to talk about getting help. Type man ls. This opens up a manual (man) page. You can also type man ls in your web browser; this will likely take you to a page like this, in case you want to keep the help page open. All man pages will have roughly the same structure: a synopsis, that lists all the possible options, a brief description, a full description of each option, some metainformation, and then a list of related commands. Can you see what the -l and -a options we used at the beginning of this tutorial do?

For this course, I will assume that you will be using man, or something like it, to learn about C library calls. There will be times that there are useful library calls that we will not have time to mention in class. Please check the man pages to learn about library and system calls and to find new calls that you need. Check man man-pages for a general tutorial on how to use the man pages, and man man as a reference for how to use the man command.

Adding to your Repository

Now, let's get you credit for the lab. Change into your t01 directory in your course repository: cd ~/209_repo/t01. Next, copy your .bashrc file into this folder, to prove that you can create an alias and set your prompt. (Hint: use the cp command. Use man to learn about it, if necessary.) Then, add this file to the repository, commit, and push: 

    git add .bashrc
    git commit
    git push

So, it may not be that simple. Depending on your machine, you might see an error that asks you to set your username or even your remote. Remember to use man to get help (or from git itself, by typing git help) and ask your TA if you get into trouble.

I find that I almost always want to push whenever I commit. You can create an alias in your .bashrc that performs a commit and a push, together, so you don't forget.

Compiling C files

One last task! For most of the rest of the term, we're going to be writing and testing C programs. Go back to your t01 directory in your repository. Use your text editor to write a hello world program. Name the file hello.c and enter the text below, which contains one error ... 

int main(int argc, char *argv[]) {
    printf("Hello world!\n");
    return 0;
}

Once you enter that code, you'll need to compile and run it. To compile it, type gcc -Wall hello.c. This will compile your hello.c file into an executable file named a.out. It uses a single argument, -Wall, which asks the compiler to emit all warnings that are generally considered important (-Wextra displays even more warnings, and -Wpedantic displays all warnings). I almost always use this argument when compiling. What do you see when you compile?

Can you fix the error? The error seems related to printf, so use man to look up the command to see if you can find the error. (Hint: printf is in section 3 of the manual, and you might find man man-pages helpful.)

Once you get the program running, modify it further so that it acts exactly as follows, based on whether an odd or even number of arguments are provided (hint: argc will be useful).: 

    alice:~/209$ gcc -Wall hello.c
    $ ./a.out
    Hello world!
    $ ./a.out a b c
    Hello 2 you, world!
    $ ./a.out 2
    Hello 2 you, world!
    $ ./a.out 2 a
    Hello world!
Ensure that the messages being printed are identical to what is displayed above, including punctuation and/or whitespace.

Once your code runs, create a .gitignore file in your git repository to ensure that executables like a.out (or any other unnecessary/temporary files) aren't accidentally added to your git repository. Commit and push the .gitignore file to your repository. Then, commit just your .c file to your repository and push it, ensuring that your a.out file is not being committed/pushed.
