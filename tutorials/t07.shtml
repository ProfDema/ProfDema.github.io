Task 1: Run the simplefork program

Open simplefork.c in your favourite editor and then read it to see what it is doing. Use the Makefile to compile and run it a few times. As you read the output, recall from the PCRS videos that it is up to the operating system to decide whether the parent or the child runs first after the fork call, and it may change from run to run.

Task 2: Fork in a loop

The program in forkloop.c takes one command-line argument that is the number of iterations of the loop that calls fork. Try it with at least 2, 3 and 4 as command-line arguments.

Question 3: How many processes are created (including the original parent) when forkloop is called with 4 as an argument?

Question 4: Use an ASCII diagram to show the processes and their relationships when you run it with the command-line argument "3". For example, the diagram for forkloop with command-line argument "2" (assuming the process id of the parent is 414 and the process ids of the children are 416, 417, and 420) might be:

    414 -> 416 -> 420
    414 -> 417

When constructing your diagram, you should not include the PID of the shell process that is the parent to your program; i.e., the left-most PID (414) in the diagram above is that of the forkloop program itself. Recall that this tutorial will be auto-tested as usual, so make sure to follow the exact format as provided above. Moreover, if any of your processes become orphaned and are adopted by another process, ignore that when constructing your diagram.

Notice that the shell prompt sometimes appears in the middle of the output. That happens because the parent process exits before some of the children get a chance to print their output.

You may also see a change in the value of the parent PID. We are getting the parent's PID from the getppid system call. If a process's parent has terminated, the process gets "adopted" by the init process which has a process id of 1. So when you see the ppid of a process is 1, that means that its parent has terminated.

Task 5: Add wait to delay the exit of the parent

Earlier, you may have seen that a parent process can terminate before its children. (If you didn't see that phenomenon, try running the forkloop program with a larger command-line argument like 5 or 6.)

However, a process can wait for its children to terminate. (Normally, when you run a program from the shell, have you noticed that the prompt doesn't appear until the program terminates?) If a process wants to wait until all its children have terminated, it needs to call wait once for each child. Add the appropriate wait calls to both childcreates and parentcreates to ensure that each parent does not terminate before its children.

Your programs should call wait as late as possible. If the process has other work to do, like creating more children, it should create the children before calling wait. If you call wait too early, then you end up slowing down your program because you are "serializing it" (reducing the number of things that run simultaneously, in parallel).
