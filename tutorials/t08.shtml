Tutorial 8: Pipes, Fork, Exec, and Dup2

Introduction

In any application that requires a user to login, the user ID and password must be validated. One approach to validation is to hand the task off to a separate process. The data is transferred to the new process over stdin; we wouldn't want to provide a password as a command-line argument, since that has a number of security risks, e.g., the password would be visible to programs such as ps that read the contents of the /proc directory (recall Assignment 2), and the password would also be visible in your .bash_history file (which keeps a history of the commands that you run in your bash terminal).

For this tutorial, you have been given a program called validate that validates user ID and password. (Check your git repository for the source code.) Your task is to apply what you've learned about processes and pipes to create a program that runs this validation program in a child process and reports the result of the validation.

Step 1: Study validate.c

Spend some time reading, compiling, and playing with the validate program. Here are some questions that you'll need to answer as you explore the code:

  • How does validate get input?
  • How many bytes does validate expect to receive on each call to read?
  • Does validate expect end of line characters? Null terminated strings?
  • How does validate report whether the username and password it has received are valid?

The questions above are just to help you understand - no need to actually submit your answers.

Step 2: Create checkpasswd.c

Your task is to complete checkpasswd.c, which reads a user ID and password from stdin, creates a new process to run the validate program, sends it the user id and password, and prints a message to stdout reporting whether the given user id-password pair is valid.

Your program should print one of three messages (which are defined in your starter code as constants):

  • "Password verified\n" if the user id and password match.
  • "Invalid password\n" if the user id exists but the password does not match.
  • "No such user\n" if the user id is not recognized (or is invalid).

Any password (or user name) that violates the character limit is considered invalid. Any such entries in the pass.txt file provided in your starter code are meant as test cases for you to try when testing your program. Note that the given password file contains a few entries that are invalid. Those examples are expected to fail, but you may find them useful for testing.

You may not use popen in your solution. The goal is to practice with fork and wait, exec, pipe, and dup2.

Hints

As a refresher, consider the contrasting behaviour of fork versus the exec family of functions:

  1. fork creates a new process (which is a child of the original process). The child process will be nearly identical to its parent, with the exception of having a different PID and return value from the fork invocation.
  2. The exec family of functions replaces the existing process image with that of a new program, but no new process is created. In other words, the existing process turns into a completely different program, but retains the same PID as the previously-running program.
  3. Both in the case of (1) a process that loads a new program into memory as a result of exec, or (2) a child process forked from a parent, the operating system allows the process to inherit all open file descriptors. In other words, if a process calls exec to load a new program into memory, the new program will have access to all the open files that the previously-running program had access to.
  4. The above point does not hold true for popen (which, again, you shouldn't use in this tutorial), which is used for loading a new program into a completely new process.

So, considering the above points, it shouldn't be surprising that if, for example, process A redirects stdout to a file and subsequently invokes a new program with exec, any printf statements in the new program will have their output redirected to that file instead of showing up on your screen.

Finally: For usage of the exec functions, you may want to refer to (1) PCRS, (2) the last 6 pages of the Week 7 slides, and (3) man exec.
