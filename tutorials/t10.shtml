Tutorial 10: Sockets

Introduction

The purpose of this exercise is to practice using the socket-related system calls. This tutorial will serve as the first milestone for completing Assignment 4. In other words: all the code that you write for this tutorial will be directly re-used for Assignment 4. It's important that you complete it, and once you do, you can be confident that you are making the expected progress on the last assignment.

In this tutorial, we will be managing a single connection at a time, so we will not be using select. That is the topic for next week's lab.

The system calls and Unix programming concepts you will be using in this exercise are:

  • Creating a socket with the socket system call
  • On the server-side: binding the socket to a network address and port to listen for and accept incoming connections using the bind, listen, and accept system calls.
  • On the client-side: connecting to a server using the connect system call.
  • Reading and writing from the socket using the read and write system calls (exactly as we did before with files and pipes).
  • Tearing down the socket using the close system call (also the same as what we did previously, with files and pipes).

Sending and Receiving Messages

Open randclient.c. It is a client that sends multiple copies of the same message to a server. Each message ends with a network newline: "\r\n".

You might expect that a server using read will receive one complete line of text with each read call. However, this isn't guaranteed to happen. For example, in a busy network or with a slow client, one line of text might be split across multiple TCP segments. A server cannot do a single read and expect to get any sensible unit of communication.

To simulate this, randclient.c artificially splits up its copies of the messages over multiple write calls. This will cause trouble for any server that expects to read entire messages at a time. readserver.c is one such server. Run readserver & (to run the server in the background) and then run randclient 127.0.0.1. Notice that the server thinks that every small piece of a message is a complete message, which is incorrect. Your task is to add buffering to your server so that it waits for a complete message (ending with the network newline) before printing the message.

Once you're done exploring how readserver works, don't forget to terminate it.

Task 1: A Buffering Server

In bufserver.c, we have given you the skeleton for a buffering server. The Step comments in the code indicate what you should do to complete the server. The design sketched out in the starter code is one solution to the problem of identifying when a message ends. Other options include sending the length of the message as an int before sending the message itself or using fixed-length messages. (For this lab and the assignment, however, you should use the protocol outlined in the starter.)

The main idea of the design we're using this week is that you have a buffer (an array) where you store pieces of a message until you receive a network newline ("\r\n"). Each message piece is appended to the data that you've already placed in the buffer. When you find a network newline, you know that you have a complete message, so you can print that message and then shift any remaining bytes (i.e., any bytes that were received after the network newline) to the front of the buffer as the start of the next message.

All of your new code should be in socket.c. Once you finish the steps, try running bufserver and then run randclient to connect to the server. The server should have one line of output per message. If there is extra garbage printed or other errors, carefully go over the steps again—it's very easy to get off-by-one mistakes here. Another tip is to be careful to note when you are guaranteed to have a null-terminated string in buf, and when you aren't.

For this part, you are asked to test your server using a simple Python client. The main objectives we will achieve by doing this are:

  • Better understand that TCP sockets work the same way, regardless of which programming language you use to create them. So it is no problem at all for a C program and Python program to communicate over a TCP connection.

Task 3: A TCP Client and Server in Python

In the client window, try:

>>> s.connect(("localhost",PORT)) # Replace PORT with your actual port number

In the server window, try: 

Python 3.6.9 (default, Nov  7 2019, 10:44:02)
[GCC 8.3.0] on linux
Type "help", "copyright", "credits" or "license" for more information.
>>> import socket
>>> s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
>>> s.bind(("localhost",PORT)) # Replace PORT with your actual port number
>>> s.listen(1)
>>> s2 = s.accept()
>>> s2[0].recv(5)
