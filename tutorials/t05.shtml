Tutorial 5: Binary I/O

Introduction

The purpose of this lab is to get a working understanding of how to use binary IO: how to read binary data from files. This lab will also provide some practice with structs. In some sense, this lab echoes lecture: we have (or will) do a very similar exercise with a different file format.

This week's lab has starter code, so start with a git pull in your repository. You should see a new directory named t05, containing the starter code and a Makefile (be sure to use this to compile your code, since that's what we will be using when marking your submission).

Background: Bitmap Images

A 24-bit bitmap file is a simple image file format that consists of three main parts:

  1. File metadata.
  2. A pixel array that stores numbers corresponding to the blue, green, and red colour values of each pixel in the image (you may recall from high school science class that red, green, and blue are the three primary colours of light, from which all other colours can be derived by additive mixing). Each colour is stored as a number with a value between 0 and 255.
  3. Optionally, the path to a "colour profile." For the purposes of this lab, you will ignore this section.

At fixed locations in the file metadata, there are three important integers, each stored using exactly 4 bytes:

  1. At byte offset 10-13, the offset in the bitmap file where the pixel array starts.
  2. At byte offset 18-21, the width of the image, in pixels.
  3. At byte offset 22-25, the height of the image, in pixels.

Suppose our bitmap image has height M and width N. We'll always assume in this lab that the width N is a multiple of 4, which simplifies the byte layout in the file. For this image, the pixel array stores exactly 3 x N x M bytes, in the following way:

  1. Each group of 3 bytes represents a single pixel, where the bytes store the blue, green, and red colour values of the pixel, in that order.
  2. Pixels are grouped by row. For example, the first 3 x N bytes in the pixel array represent the pixels in the top-most row of the image.

That's all you need for this lab, but if you're curious about learning more about the bitmap file format, you can start with this Wikipedia page.

Reading Binary Data

The starter code file bitmap_printer.c opens a bitmap for processing, and the file bitmap.c contains functions for working with bitmaps. Your first task is to use the description of the bitmap metadata to implement the function read_bitmap_metadata.

Don't just try to read in the entire bitmap file, byte by byte. Use fseek to navigate to different byte locations so that you only read the three required integers. Then, you can use fread to read the expected number of bytes into a variable of the type you want.

Two bitmaps are included in the starter files for this week. You can use these files for testing. You can also use your own bitmaps, but it could be tricky. There are many variations of this file format, and we're only covering the 24-bit version in this tutorial, so you might need to convert new images to this format yourself (and then crop the image to ensure the width is a multiple of 4).

Reading Binary Arrays

Next, use the metadata read by read_bitmap_metadata to read all of the pixels. Place the pixel data into a two-dimensional array on the heap. bitmap.c contains a large comment explaining how to approach this problem. Note that because each pixel colour value (blue, green, or red) is stored in just one byte, the type we use is unsigned char: a 1 byte value that represents numbers from 0 to 255 (inclusive).

If the sizeof(struct pixel) == 3 on your system (check this!), you can use a single, large fread to get an entire row of data. If it does not, you'll need a loop that reads one struct at a time.

Also, you'll need to remove a return from bitmap_printer.c to allow your function to be called.

Free Your Memory

Finally, add some code to the bottom of the main function in bitmap_printer.c to close the bitmap file and free all of the program's dynamically-allocated memory. Use valgrind to check that no heap memory is being leaked.

Valgrind and Address Sanitizer:: If you attempt to use Valgrind and Address Sanitizer together, Valgrind will be unable to track heap allocations. You may see errors along the lines of ASan runtime does not come first in initial library list.... So, to get a complete Valgrind report you will need to temporarily remove the -fsanitize=address compiler flag.

The easiest way to go about this would be to make use of the Address Sanitizer to locate any bugs that it is able to detect. Then, disable Address Sanitizer and use Valgrind that you do not have any other errors (such as reads on uninitialized variables).
